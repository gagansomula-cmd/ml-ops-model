# IAM Role-Based Authentication Architecture

This document explains how the ML-Ops infrastructure uses **IAM roles instead of hardcoded credentials**.

## 🔐 Authentication Flow

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow Trigger (push to main)              │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ GitHub OIDC Provider                                         │
│ - Verifies workflow authenticity                             │
│ - Issues temporary token with repository context             │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ AWS STS AssumeRoleWithWebIdentity                           │
│ - Exchanges GitHub token for AWS temporary credentials      │
│ - Credentials valid for ~1 hour                             │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ GitHubActionsRole (IAM Role)                                │
│ - Terraform provisioning                                     │
│ - EKS cluster management                                     │
│ - S3 model uploads                                           │
│ - Kubernetes cluster configuration                           │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ AWS Services                                                 │
│ - Terraform (infrastructure provisioning)                    │
│ - EKS (Kubernetes cluster)                                   │
│ - S3 (model storage)                                         │
│ - IAM (role management)                                      │
└─────────────────────────────────────────────────────────────┘
```

## 📋 IAM Roles & Policies

### 1. GitHubActionsRole

**Purpose**: Used by GitHub Actions CI/CD pipeline

**Permissions**:
- `eks:*` - Full EKS cluster management
- `ec2:*` - VPC, security groups, networking
- `s3:*` - S3 bucket management
- `iam:*` - IAM role and policy management (with IAMFullAccess)
- `autoscaling:*` - Auto-scaling group management
- `elasticloadbalancing:*` - Load balancer management
- `logs:*` - CloudWatch logs

**AWS CLI Command**:
```bash
aws sts get-caller-identity
# Returns: GitHubActionsRole
```

**Location**: `terraform/github-oidc.tf`

### 2. EKS Node Role (IAM role for EC2 worker nodes)

**Purpose**: Used by Kubernetes pods running on worker nodes

**Permissions**:
- `AmazonEKSWorkerNodePolicy` - Node operations
- `AmazonEKS_CNI_Policy` - Networking
- `AmazonEC2ContainerRegistryReadOnly` - ECR access
- `eks-s3-models-access` - S3 model access (custom policy)

**Used by**: KServe pods downloading models from S3

**Location**: `terraform/eks.tf` and `terraform/s3-models.tf`

### 3. OIDC Provider (GitHub)

**Purpose**: Trust relationship for GitHub Actions

**Configuration**:
- URL: `https://token.actions.githubusercontent.com`
- Condition: `repo:your-repo:ref:refs/heads/main`
- Audience: `sts.amazonaws.com`

**Location**: `terraform/github-oidc.tf`

## 🔄 Kubernetes Access - aws-auth ConfigMap

The `aws-auth` ConfigMap bridges AWS IAM and Kubernetes RBAC:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: aws-auth
  namespace: kube-system
data:
  mapRoles: |
    # EKS Node Role → Kubernetes system:nodes group
    - rolearn: arn:aws:iam::ACCOUNT:role/ml-ops-cluster-eks-node-role
      username: system:node:{{EC2PrivateDNSName}}
      groups:
        - system:bootstrappers
        - system:nodes
    
    # GitHub Actions Role → Kubernetes system:masters group
    - rolearn: arn:aws:iam::ACCOUNT:role/GitHubActionsRole
      username: github-actions
      groups:
        - system:masters
```

**What this does**:
1. Maps AWS IAM roles to Kubernetes users
2. Assigns Kubernetes groups for RBAC
3. `system:masters` = full admin access

**Location**: Generated by `terraform/eks.tf` → `aws-auth-configmap.yaml`

## 🚀 Credential Flow by Component

### GitHub Actions Workflow

```
GitHub Action (with OIDC)
    ↓
GitHub OIDC Token
    ↓
AWS STS AssumeRoleWithWebIdentity
    ↓
GitHubActionsRole (temporary credentials)
    ↓
Terraform / AWS CLI / kubectl
    ↓
AWS Services (EKS, S3, IAM, etc.)
```

**Key Points**:
- ✅ No hardcoded AWS keys stored in repository
- ✅ Temporary credentials (auto-expiring)
- ✅ CloudTrail audit trail (who did what, when)
- ✅ GitHub Actions → AWS trace (OIDC token)

### KServe Pods (S3 Model Access)

```
KServe Pod (running on EKS node)
    ↓
EC2 Instance Metadata Service (IMDSv2)
    ↓
EKS Node IAM Role (temporary credentials)
    ↓
S3 Bucket (model download)
```

**Key Points**:
- ✅ Pod inherits node role permissions
- ✅ Credentials auto-refreshed by IMDS
- ✅ No hardcoded S3 credentials in pod

### prepare_model.py Script

```
Script (using boto3)
    ↓
Detect credentials from:
  1. GitHub Actions IAM role (CI/CD)
  2. Instance profile (AWS EC2/EKS)
  3. AWS CLI configuration (local)
    ↓
boto3 S3 Client
    ↓
S3 Upload (using IAM role permissions)
```

**Key Points**:
- ✅ No explicit AWS_ACCESS_KEY_ID needed
- ✅ boto3 automatically uses available credentials
- ✅ Works in GitHub Actions, EC2, EKS, and local AWS CLI

## 🛡️ Security Best Practices Applied

### 1. No Hardcoded Credentials ✅
- ❌ AWS_ACCESS_KEY_ID in secrets
- ❌ AWS_SECRET_ACCESS_KEY in repository
- ✅ OIDC for GitHub Actions
- ✅ Instance profiles for EC2/EKS

### 2. Least Privilege ✅
- GitHub Actions role has only needed permissions
- Node role has only S3 read/write permissions
- S3 policy limited to specific prefixes (`linear-regression/*`, `models/*`)

### 3. Temporary Credentials ✅
- OIDC tokens valid for short period (~1 hour)
- STS credentials auto-refreshed
- CloudTrail audit logging available

### 4. RBAC in Kubernetes ✅
- aws-auth ConfigMap controls Kubernetes access
- GitHub Actions role mapped to `system:masters`
- Can revoke access by updating ConfigMap

### 5. Encrypted in Transit ✅
- All AWS API calls use HTTPS/TLS
- S3 transfers encrypted
- kubectl uses HTTPS to API server

## 🚨 Troubleshooting

### "Unauthorized" when running kubectl

**Problem**: `error: You must be logged in to the server (Unauthorized)`

**Solutions**:
1. Check aws-auth ConfigMap is applied:
   ```bash
   kubectl get configmap aws-auth -n kube-system
   ```
2. Verify GitHubActionsRole is in mapRoles
3. Re-apply ConfigMap:
   ```bash
   kubectl apply -f aws-auth-configmap.yaml
   ```

### S3 Upload Fails in GitHub Actions

**Problem**: `Access Denied` when uploading model to S3

**Solutions**:
1. Check GitHub Actions role has s3-models-access policy:
   ```bash
   terraform output github_actions_role_arn
   ```
2. Verify policy attached:
   ```bash
   aws iam list-attached-role-policies --role-name GitHubActionsRole
   ```
3. Re-apply Terraform:
   ```bash
   cd terraform && terraform apply
   ```

### KServe Can't Download Model from S3

**Problem**: Pod fails to download model with access error

**Solutions**:
1. Check node role has eks-s3-models-access policy
2. Verify S3 bucket and model path correct
3. Check pod's IAM role:
   ```bash
   kubectl describe pod -n kserve -l app=sklearn-model
   # Check for role ARN annotation
   ```

## 🔍 Key Takeaways

1. **No Hardcoded Credentials**: AWS access keys never stored in repository
2. **OIDC Federation**: GitHub Actions authenticates via GitHub OIDC provider
3. **Temporary Tokens**: All credentials are temporary and auto-expiring
4. **IAM Roles**: All AWS access via IAM roles (least privilege)
5. **aws-auth ConfigMap**: Maps IAM roles to Kubernetes permissions
6. **Audit Trail**: CloudTrail tracks all API calls by role
7. **Zero Trust**: Each component authenticates separately with IAM roles

```
└──────────┬──────────┘
           │
           ↓
┌─────────────────────┐
│ AWS STS             │ ← Validates OIDC token

           │

           │
           ↓
    ┌──────────────────────────────┐

    └──────────────────────────────┘
```

This architecture ensures **maximum security with zero hardcoded secrets**! 🔐
